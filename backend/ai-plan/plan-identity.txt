# Plan: Identity Module — Supabase + Prisma Integration (v3)

## TL;DR
Build identity module following existing products module patterns. Supabase handles authentication (login, register, sessions, JWT). Prisma handles authorization data (roles, permissions, profiles) in `identity` schema. **Identity DB lives on Supabase Postgres** (local CLI in dev, cloud project in prod) — separate from Docker Postgres used by core/products. Supabase RLS enabled. `SUPABASE_PUBLISHABLE_KEY` (not legacy anon key). Prisma schema requires `directUrl` for Supabase cloud connection pooling.

---

## Architecture: Two Postgres Instances

| Environment | core + products DB | identity DB | Auth |
|-------------|-------------------|-------------|------|
| **Local dev** | Docker Postgres `:5432` | Supabase CLI Postgres `:54322` | Supabase CLI local |
| **Production** | Docker/cloud Postgres | Supabase cloud Postgres (Supavisor pooler) | Supabase cloud |

Key implication: Makefile `schemas`/`migrate`/`generate` commands for identity target **different Postgres** than core/products.

### Step 1.6 — Supabase Cloud: Custom Prisma User *(one-time, manual)*
In Supabase SQL Editor, create dedicated prisma user (per official docs):
```sql
CREATE USER "prisma" WITH PASSWORD 'custom_password' BYPASSRLS CREATEDB;
GRANT "prisma" TO "postgres";
GRANT USAGE ON SCHEMA identity TO prisma;
GRANT CREATE ON SCHEMA identity TO prisma;
GRANT ALL ON ALL TABLES IN SCHEMA identity TO prisma;
GRANT ALL ON ALL SEQUENCES IN SCHEMA identity TO prisma;
ALTER DEFAULT PRIVILEGES FOR ROLE postgres IN SCHEMA identity GRANT ALL ON TABLES TO prisma;
ALTER DEFAULT PRIVILEGES FOR ROLE postgres IN SCHEMA identity GRANT ALL ON SEQUENCES TO prisma;
```

## Phase 3: Infrastructure Layer *(depends on Phase 2)*

### Step 3.1 — Prisma Schema & Client *(depends on 2.1)*
Create `libs/identity/src/infrastructure/prisma/`:

**`schema.prisma`** — CRITICAL: includes `directUrl` for Supabase connection pooling:
```prisma
generator client {
  provider = "prisma-client-js"
  output   = "./generated/prisma"
}

datasource db {
  provider  = "postgresql"
  url       = env("DATABASE_URL_IDENTITY")
  directUrl = env("DIRECT_URL_IDENTITY")
}

model UserProfile {
  id          String     @id              // Supabase Auth UUID, no @default()
  email       String     @unique
  createdAt   DateTime   @default(now())
  lastLoginAt DateTime?
  userRoles   UserRole[]
}

model Role {
  id          String           @id @default(cuid())
  name        String           @unique
  description String?
  rolePerms   RolePermission[]
  userRoles   UserRole[]
}

model Permission {
  id          String           @id @default(cuid())
  name        String           @unique
  description String?
  rolePerms   RolePermission[]
}

model RolePermission {
  roleId       String
  permissionId String
  role         Role       @relation(fields: [roleId], references: [id])
  permission   Permission @relation(fields: [permissionId], references: [id])
  @@id([roleId, permissionId])
}

model UserRole {
  userId String
  roleId String
  user   UserProfile @relation(fields: [userId], references: [id])
  role   Role        @relation(fields: [roleId], references: [id])
  @@id([userId, roleId])
}
```

**`config.ts`** — `buildIdentityPrismaConfig()` using `buildPrismaModuleConfig('identity', 'DATABASE_URL_IDENTITY')`

**`client.ts`** — `createIdentityPrisma(config?)` — pattern: products/client.ts

**`tx.ts`** — `setIdentityPrismaClient()`, `identityDb()`, `runInIdentityTx()` — AsyncLocalStorage pattern

**`di.ts`** — `IDENTITY_TOKENS`, `registerIdentityPrisma()`, `registerIdentityRepository()`

## Decisions
- Supabase Auth UUID = UserProfile.id (no mapping table)
- RLS enabled on Supabase project
- Auth in Supabase, roles/permissions in Prisma
- **Two Postgres instances**: Docker (core/products) + Supabase (identity)
- Prisma schema with `directUrl` for Supabase cloud connection pooling
- `SUPABASE_PUBLISHABLE_KEY` (NOT legacy `anonKey`)
- Custom `prisma` user on Supabase cloud with `BYPASSRLS CREATEDB` (per Supabase docs)
- Contracts in `libs/integration-contracts/`
- Dot-notation permissions (products.read)
- HTTP middleware (verifyJWT) — follow-up phase
